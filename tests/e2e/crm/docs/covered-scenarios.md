# Покриті сценарії crm

## Smoke

### 1. Авторизує через email і пароль

Кроки:

1. Відкрити `/login`
2. Перевірити, що видно поля `Email`, `Пароль` і кнопку `Увійти`
3. Ввести валідний `email` з `data/<env>/crm/auth.json`
4. Ввести валідний `password` з `data/<env>/crm/auth.json`
5. Натиснути `Увійти`
6. Перевірити, що `POST /api/v1/users/login/` повернув `200`
7. Перевірити відкриття CRM кабінету

Файл:

- `tests/e2e/crm/smoke/login.smoke.spec.ts`

### 2. Модерує замовлення через публічний доступ

Кроки:

1. Авторизуватись у CRM через email і пароль
2. Відкрити `/orders`
3. Натиснути на картку замовлення в лівому списку
4. Увімкнути чекбокс `Публічний доступ`
5. Натиснути `Підтвердити`
6. Перевірити повідомлення `Успішно збережено`
7. Перевірити повідомлення `Замовлення успішно підтверджено`
8. Дочекатись повторного завантаження сторінки
9. Перевірити, що промодерованого замовлення більше немає в списку
10. Перевірити, що після reload активна картка не обрана і права панель порожня

Fallback-поведінка:

- якщо на сторінці `/orders` немає жодної картки для модерації, тест не падає
- у логи пишеться повідомлення про відсутність ордерів для модерації
- сценарій завершується без виконання бізнес-дії

Файли:

- `tests/e2e/crm/smoke/order-moderation.smoke.spec.ts`
- `tests/_shared/helpers/crm-order-moderation.flow.ts`
- `tests/_shared/helpers/crm-orders.page.ts`

## Технічна база для наступних сценаріїв

### 3. Підготовлено locator map для crm orders

Що зафіксовано:

1. Sidebar-навігацію з `title`-локаторами для меню CRM
2. Лівий список замовлень із картками `.card-order` і активним станом `.active`
3. Праву панель деталей із формою `form.form-moderate-order`
4. Кнопки дій `Копіювати замовлення`, `Зберегти`, `Підтвердити`
5. `id`-локатори ключових полів форми, локації, оплати та змін
6. Ризикові місця з іконковими кнопками й дубльованим `id` у shift inputs

Файли:

- `tests/_shared/helpers/crm-orders.page.ts`
- `tests/e2e/crm/docs/orders-locators.md`
