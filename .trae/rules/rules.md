# Правила написання автотестів Playwright

## Обов'язковий порядок роботи

1. Спочатку пройти флоу вручну або через браузерні інструменти та зібрати актуальну інформацію:
   - видимі стани сторінки;
   - стабільні селектори;
   - переходи між кроками;
   - тексти помилок і успішних станів;
   - поведінку на `dev`, `stage`, `prod`, якщо вона відрізняється.
2. Лише після аналізу писати код тесту.
3. Після написання коду обов'язково запускати перевірки та сам тест.

## Мова

- Усі назви кроків, коментарі в тестах, логування, тексти помилок у helper'ах та службові повідомлення пишемо українською мовою.
- Якщо в UI текст іншою мовою, у тестах використовуємо фактичний текст UI лише для пошуку елемента або перевірки.

## Структура проєкту

- `tests/e2e/<area>/smoke` — тільки критичні позитивні сценарії.
- `tests/e2e/<area>/regression` — негативні сценарії, валідації, edge cases.
- `tests/e2e/<area>/docs` — обов'язкова папка з документацією по запуску і покриттю.
- `tests/_shared/helpers` — page objects, flow helpers, domain helpers.
- `tests/_shared/fixtures` — спільні fixture'и, що читають data/config.
- Не дублювати селектори між тестами: селектори живуть у page/helper класах.
- Не дублювати доступ до data-файлів у spec-файлах: доступ тільки через fixtures/loaders.
- У кожній `tests/e2e/<area>/docs` обов'язково створювати 2 файли:
  - файл з описом запуску тестів на різних оточеннях;
  - файл з описом уже покритих сценаріїв і кроків.

## Масштабування

- Один і той самий spec має працювати на `dev`, `stage`, `prod`.
- Оточення та розділ продукту визначаються тільки через `APP_AREA` і `TEST_ENV`.
- Тест не повинен містити захардкоджені URL, логіни, паролі, телефони, OTP.
- Усі тестові дані повинні братись із `data/<env>/<area>/auth.json` та `data/<env>/<area>/test-data.json`.
- Якщо контракт data-файлу змінюється, треба оновити типи, fixtures і самі data-файли одночасно.

## Стабільність

- Статичні очікування заборонені:
  - не використовувати `waitForTimeout`;
  - не використовувати власні затримки через `setTimeout`.
- Очікування тільки через стани:
  - `toBeVisible`;
  - `toHaveURL`;
  - `toHaveText`;
  - `toHaveCount`;
  - `toBeEnabled` / `toBeDisabled`;
  - інші явні очікування стану Playwright.
- Після дії має бути перевірка результату дії.
- Позитивний сценарій не можна вважати успішним лише тому, що сторінка "не /login":
  - потрібно перевіряти конкретний цільовий стан авторизованої сторінки.
- Негативний сценарій має перевіряти конкретну помилку або очікуваний стан, а не тільки факт, що URL не змінився.

## Селектори

- Пріоритет селекторів:
  1. `getByRole`
  2. `getByLabel`
  3. `getByTestId`
  4. стабільний `locator` з явним атрибутом
- Не покладатись на крихкі селектори:
  - placeholder, якщо є label або role;
  - CSS-класи, згенеровані UI бібліотекою;
  - індекси елементів без потреби.
- Якщо стабільного селектора немає, треба:
  - задокументувати ризик;
  - винести селектор у helper;
  - мінімізувати місця його використання.

## Повторюваність і helper'и

- Якщо одна й та сама послідовність дій використовується більше одного разу, її потрібно винести в helper/page object.
- Якщо одна й та сама підготовка повторюється у spec-файлах, її потрібно винести у fixture.
- Не дублювати однакові expect'и без потреби.
- Повторно використовувані перевірки успішного/неуспішного логіну повинні бути методами page object/helper.

## Дані та контракти

- Типи даних повинні відповідати реальним JSON-файлам.
- Заборонено "тихо" кастити JSON до типу, якщо контракт може розійтись.
- Для критичних data-файлів бажана runtime-перевірка структури перед використанням.
- Якщо для певного середовища немає валідних даних, тест не повинен мовчки падати без пояснення:
  - має бути зрозуміла причина в логах або помилці.

## Якість коду

- Перед завершенням роботи обов'язково запускати:
  - `npm run check`
- Якщо можна авто-виправити:
  - `npm run fix`
- Перед комітом і пушем перевірки мають проходити через husky hooks.

## Оформлення тестів

- Назви `describe` і `test` повинні бути зрозумілими, короткими та відображати бізнес-сенс.
- Один тест перевіряє один сценарій.
- Коментарі додаємо лише там, де без них важко зрозуміти бізнес-крок.
- Усі нові тести повинні одразу відповідати цим правилам.

## Обов'язкова пам'ятка перед написанням тесту

- Спочатку прочитай цей файл.
- Потім перевір UI вручну або браузерним інструментом.
- Потім напиши helper/fixture, якщо це повторюваний флоу.
- Потім напиши тест.
- Потім запусти `npm run check` і сам тест.
