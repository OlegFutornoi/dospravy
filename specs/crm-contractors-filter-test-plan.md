# План покриття: core-перевірки фільтра кандидатів (crm)

## Мета

- Довести, що фільтр на сторінці `/contractors` працює коректно
- Перевірити не лише зміну списку, а правильність даних у результатах
- Залишити тільки мінімальний набір перевірок, якого достатньо для висновку, що фільтрація працює вірно

## Scope Фільтрів

- У межах цієї задачі враховуються тільки фільтри, які видно на скріншоті панелі `Фільтри`
- До scope входять тільки:
  - `Місто`
  - `Спеціалізації`
  - `Джерело походження`
  - `Агент`
  - `Договір`
  - `Акаунт`
- Поле `Пошук...` не є частиною scope фільтрів у цій задачі і використовується лише як технічний спосіб ізоляції seed-даних для автотестів

## Що саме мають довести тести

- після вибору фільтра в request іде правильний параметр
- backend повертає тільки ті записи, які відповідають умові
- UI показує тільки релевантні записи
- комбінований фільтр працює як логічне `AND`
- `Очистити всі` повністю прибирає активні умови

## Мінімальний набір тестових даних

- Достатньо `4` контрольованих кандидатів
- Поля, які обов'язково мають відрізнятись:
  - `Місто`
  - `Спеціалізація`
  - `Джерело`

Приклад seed-набору:

- кандидат `A`: `Київ` + `Спеціалізація 1` + `Джерело 1`
- кандидат `B`: `Київ` + `Спеціалізація 2` + `Джерело 2`
- кандидат `C`: `Львів` + `Спеціалізація 1` + `Джерело 2`
- кандидат `D`: `Львів` + `Спеціалізація 2` + `Джерело 1`

Цього набору достатньо, щоб перевірити:

- одиночний фільтр по місту
- одиночний фільтр по спеціалізації або джерелу
- комбінований фільтр `місто + спеціалізація`
- відсутність зайвих записів у результаті

## Core UI Сценарії

### 1. UI: фільтр по місту

1. Авторизуватись у CRM.
2. Відкрити `/contractors`.
3. Відкрити панель `Фільтри`.
4. Обрати місто, наприклад `Київ`.
5. Натиснути `Застосувати`.
6. Перехопити request на список кандидатів.
7. Перевірити, що request містить параметр вибраного міста.
8. Перевірити, що у таблиці залишились тільки кандидати з міста `Київ`.
9. Перевірити, що кандидати з міста `Львів` відсутні.

Критерії успіху:

- у request передається правильне значення фільтра;
- усі знайдені записи відповідають місту `Київ`;
- жоден нерелевантний запис не потрапляє в результат.

### 2. UI: фільтр по джерелу

1. Відкрити панель `Фільтри`.
2. Обрати `Джерело`
3. Натиснути `Застосувати`.
4. Перевірити request.
5. Перевірити, що кожен запис у таблиці відповідає вибраному значенню.
6. Перевірити, що записи з іншими значеннями відсутні.

Критерії успіху:

- фільтр коректно працює не лише по місту, а й по ще одному полю з панелі фільтрів;
- результати відповідають очікуваним даним.

### 3. UI: комбінований фільтр `місто + джерело`

1. Відкрити панель `Фільтри`.
2. Обрати `Місто = Київ`.
3. Обрати `Джерело = Джерело 1`.
4. Натиснути `Застосувати`.
5. Перевірити request.
6. Перевірити, що у списку залишився тільки кандидат, який одночасно відповідає обом умовам.
7. Перевірити, що:
   - кандидат з тим самим містом, але іншим джерелом відсутній;
   - кандидат з тим самим джерелом, але іншим містом відсутній.

Критерії успіху:

- комбінований фільтр працює як `AND`;
- у видачу не потрапляють частково релевантні записи.

### 4. UI: `Очистити всі`

1. Застосувати один або два фільтри.
2. Переконатись, що список змінився.
3. Натиснути `Очистити всі`.
4. Перевірити, що всі поля очищені.
5. Перевірити, що новий request не містить активних параметрів фільтра.
6. Перевірити, що список повернувся до baseline-стану.

Критерії успіху:

- фільтр очищається повністю;
- після очищення не лишається прихованих умов.

## Core API Сценарії

### 1. API: одиночний фільтр

1. Авторизуватись у CRM через API.
2. Виконати `GET /users/crm/contractors/` з одним фільтром, наприклад по місту.
3. Перевірити код відповіді.
4. Перевірити, що кожен запис у `results` відповідає вибраному місту.
5. Перевірити, що записи з іншим містом не повертаються.

Критерії успіху:

- backend правильно застосовує одиночний фільтр;
- відповідь містить тільки релевантні дані.

### 2. API: комбінований фільтр

1. Виконати `GET /users/crm/contractors/` з двома параметрами, наприклад `місто + джерело`.
2. Перевірити код відповіді.
3. Перевірити, що кожен запис відповідає обом умовам одночасно.
4. Перевірити, що частково релевантні записи відсутні.

Критерії успіху:

- backend застосовує комбіновані умови як `AND`;
- відповідь не містить зайвих записів.

## Мінімальний висновок після проходження цього набору

- фільтр передає правильні умови в backend
- backend фільтрує дані правильно
- UI відображає тільки правильні результати
- комбінована логіка працює правильно
- очищення не залишає активних фільтрів

## Порядок реалізації

1. Підготувати контрольований seed з `4` кандидатів
2. Зняти фактичні query params з мережі
3. Реалізувати `UI` тест по місту
4. Реалізувати `UI` тест по джерелу
5. Реалізувати `UI` комбінований тест
6. Реалізувати `UI` тест `Очистити всі`
7. Реалізувати `API` тест з одиночним фільтром
8. Реалізувати `API` тест з комбінованим фільтром

## Карта змін для реалізації

### Обраний core-набір для реалізації

- `UI`: фільтр по `Місто`
- `UI`: фільтр по `Джерело походження`
- `UI`: комбінований фільтр `Місто + Джерело походження`
- `UI`: `Очистити всі`
- `API`: одиночний фільтр по `Місто`
- `API`: комбінований фільтр `Місто + Джерело походження`

Причина вибору:

- `Місто` і `Джерело` стабільно видно і в таблиці, і в `API` відповіді
- для `Джерело` вже є явний мапінг UI значень у `API` enum-и
- ця комбінація дає надійне мінімальне покриття без залежності від складніших полів

### Правило для seed-даних

- кожен набір із `4` кандидатів створюється з одним спільним унікальним токеном у ПІБ
- перед перевіркою фільтра використовується `search` по цьому токену
- це ізолює автотест від уже існуючих кандидатів у середовищі
- після цього фільтр перевіряється тільки на контрольованому наборі записів
